# Prompt Writer Skill

Generate detailed AI personalization prompts based on campaign strategy AI variables. These prompts instruct Ken AI's AI to create personalized content for each prospect.

## Parser Contract

`prompts.md` is consumed by the `campaign-configuration` skill's plan parser. See [parser-contract.md](../campaign-configuration/references/parser-contract.md) for the full spec. Hard rules this skill must honor:

- `## User Prompts (To output: true)` H2 heading - exact phrasing. `## User Prompts` alone still parses but warns.
- One H3 per variable using Title Case (`### First Line`, not `### first_line`). H3 name is ≤30 chars.
- Every H3 must contain a `**Prompt**:` marker followed by the prompt text. No marker → variable is silently dropped (hard error if the emails reference it).
- Prompt text ≤4096 chars. Runs until next `### `, `## `, or `---`. Do NOT use `---` inside a prompt body - it cuts the prompt short.
- Variable names must match exactly what `emails_v2.md` uses in `[Title Case]` brackets.
- Optional `## Rewriting Instructions` H2 at the bottom - its body is pushed as the Flow.Rewriting steering paragraph.

## Plan Folder Resolution

This skill receives a `{segment_folder}` path (e.g., `{workspace}/03-15 - plan 1/1 - early-stage/`). To access plan-level files like `configuration.json`, resolve `{plan_folder}` as the **parent directory** of `{segment_folder}`.

## Output Format

### How the Backend Works

The backend auto-generates system prompts from campaign data:
- **Instructions**: Generated from `Constants.DefaultInstructions` + `Client.ContactName` + `Client.CompanyName`
- **Company Context**: Generated from `Client.CompanyContext`
- **Rules**: Generated from `Constants.DefaultRules`
- **Email Sequence**: Generated from the campaign's `EmailSequence` field
- **Rewriting**: Uses `Constants.DefaultRewritingPrompt` as fallback if no user-defined rewriting prompts exist

CSMs only need to write **user prompts** (`to_output: true`) that generate personalized variables.

### Prompt Naming Convention
Prompt names must be short, plain text with Title Case. No underscores, colons, or special characters.
- Good: "First Line", "Subject Line", "PS Line"
- Bad: "personalized_first_line", "campaign_name:company_context", "subject_line"

Generate `prompts.md` with this structure:

```markdown
# AI Personalization Prompts

Campaign: [campaign folder name]
Generated: [date]

## Variables Overview

| Variable | Location | Email(s) | Rewrite |
|----------|----------|----------|---------|
| [name] | [location] | [1/2/3/all] | [yes/no] |

## User Prompts (To output: true)

These prompts generate the actual AI variables that appear in the copy:

### [Prompt Name]
**Location**: [prompt_location]
**Email(s)**: [email_number or "all"]
**Rewrite**: [yes/no]

**Prompt**:
[Full detailed prompt text]

---

[Repeat for each variable]

## Rewriting Instructions

[2-10 sentence steering paragraph that tunes voice, tone, or sequence flow for this campaign. Omit this H2 section entirely if the campaign has no distinctive voice/flow requirements. Do NOT duplicate the backend's default rules - focus only on what makes THIS client/campaign distinctive.]
```

The `## Rewriting Instructions` H2 is parsed by the campaign-configuration skill's plan parser and pushed as a `Flow.Rewriting` system prompt via the prompt library API. The backend appends it after the default rewriting prompt, spam rules, formatting rules, company context, audience description, and email sequence context - so you only need to write the delta.

## Prompt Writing Guidelines

Each prompt must specify:

### 1. What to Generate
Clear instruction on the output:
- "Write a subject line that..."
- "Generate a 1-sentence opening that..."
- "Create a pain point reference that..."

### 2. Tone and Style
How it should sound:
- "Conversational and curious"
- "Professional but friendly"
- "Insightful, not generic"

### 3. Constraints
Length and format rules:
- "Under 50 characters"
- "1-2 sentences max"
- "No questions or emojis"
- "Start with 'PS'"

### 4. Fallback Strategy
What to do if data is missing:
- "If not available, use their role instead"
- "Skip if no relevant data found"

## Standard Variable Templates

### Subject Line
```
Write a very short subject line that grabs attention and sparks curiosity in fewer than 6 words, ideally 3-4.
Make it relevant to a challenge, wish, hobby, or activity. Do not add emojis or punctuation. Use lowercase except for the first letter, and capitalize the first letter of company and person names.
Focus on a single, specific small detail from your research; it can be personal. The goal is to intrigue the prospect quickly and make them curious about the email's contents in just 3-4 words.
```

### First Line
```
Write the first line of the email. Refer to the template above to see where it fits in the initial message. Don't write any greeting.
The goal is to hook their attention, make the line very personalized, and lead smoothly into the rest of the email. Keep this line short, under 20 words.
It should spark curiosity, be intriguing and simple, and avoid gimmicks. Do not sound robotic. Most important: do not be boring.
Avoid emojis, exclamation marks, and questions. Don't just summarize information about them.
Aim for a pleasant, genuine tone that reflects the company: smart, concise, personalized, and straightforward, using simple language.
```

### PS Line
```
Write a PS line for this email that starts with "PS" and continues. Keep it short: one or two sentences. The goal is to be human and to reflect your personality as a creator. This line should be casual and personal - an opportunity to add more humanity to your outreach.
Make the PS about them, not about you. Include a very small personal detail that few people notice - something hidden on their website or LinkedIn profile. Don't try too hard or make broad assumptions; if you do assume something, mention that you're assuming. Humor is good but subtle - if you see something funny in their profile, you can laugh at it or lightly develop the joke.
This is also a chance to share a small, catchy idea for them. Keep it genuine and brief. Do not invent connections, invite them to anything, or offer help you can't back up. Avoid generic compliments; instead, make them feel noticed and special.
```

### Role Pain Point
```
Based on their job title from LinkedIn, reference a pain point typical for that role. Be specific to seniority: VPs/Directors - scaling, team efficiency, cross-functional alignment. C-suite - strategy, competitive positioning, board concerns. Managers - execution, reporting, resource constraints. 1 sentence max that feels like something only someone in their role would understand.
```

### Industry Context
```
Based on company industry from LinkedIn, reference a common challenge or priority in that industry. Keep evergreen - not news or events, just general industry dynamics. Example for healthcare: 'In healthcare, compliance and patient data security are always top of mind.' 1 sentence, conversational.
```

### Company Size Context
```
Based on LinkedIn company headcount, reference challenges typical to that size. 10-50: wearing multiple hats, limited resources. 50-200: scaling pains, process gaps. 200-1000: coordination complexity, tool sprawl. 1000+: enterprise complexity, change management. 1 sentence showing you understand their scale.
```

## Prompt Customization Rules

**CRITICAL RULES** for customizing the standard templates above:

### Examples Policy
- **DO NOT add examples** for: First Line, PS Line, Subject Line, unless they're heavily customized
- These types of variables should feel natural and unrestricted - examples constrain creativity
- **CAN add examples** for: Other custom prompts that need specific structure/format (e.g., industry-specific variables, complex body text)

### PS Line Guidelines
- **Use the DEFAULT PS line template** and only modify SLIGHTLY for campaign context
- Never create completely custom PS line prompts from scratch
- **Format**: Always use "PS: " (with colon and space, NOT "PS -" or "PS." or just "PS")
- Modifications should be minor adjustments, not rewrites

### Subject Line Guidelines
- **Keep subject lines close to default formula**, only customize for campaign-specific angle
- Never add "Quick question" or typical outreach messaging to subject line prompts
- Subject line prompt changes should be SLIGHT to match email sequence or client tone
- Avoid sales clichés in subject line instructions

### What You CAN Customize
- Data source references (which LinkedIn fields to prioritize)
- Tone adjustments (more casual, more professional)
- Industry-specific context or pain points
- Length constraints (shorter or longer)

### What You Should NOT Customize
- The core structure of First Line, PS Line, or Subject Line templates
- Adding example outputs that constrain the AI
- Changing the fundamental approach of these standard variables

## Rewriting Instructions

AI Rewriting runs on every campaign. The backend auto-injects a rich rewriting system prompt: core principles, email flow integration, style rules, spam/deliverability rules, formatting rules, company context, audience description, and the full email sequence. You do NOT need to recreate any of that.

Your job is to write a short, optional **steering paragraph** (2-10 sentences) that tunes the rewriter for this specific client/segment. This gets appended to the backend's defaults as "User Custom Rewriting Instructions" and is marked as taking priority over the default prompt.

### The `to_rewrite` field
`to_rewrite` on each user prompt controls whether its output gets passed through the rewriter:
- `to_rewrite=0` - NOT rewritten (use for subject lines, short outputs)
- `to_rewrite=1` - WILL be rewritten (use for First Line, PS Line, body content)

### What to Write About

Pick 1-3 of these angles based on what's distinctive about this client/segment:
- **Voice & tone** - How should sentences sound? (punchy, technical, warm, dry, founder-to-founder, etc.)
- **Sentence structure & rhythm** - Short/long, fragments OK, pacing, whether to mirror the client's own writing style
- **Sequence flow** - How personalized lines should hand off into this segment's specific body copy
- **Campaign-specific anti-patterns** - Phrases or constructions to avoid for this client (e.g. "don't sound like a recruiter" for an HR-tech client)
- **Research signaling** - How much the rewrite should show that research was done

### What NOT to Write

- Generic rules already in the default prompt (no em-dashes, no spam words, no AI-tells like "I noticed", no exclamation marks, etc.) - the backend handles these
- Formatting instructions - the backend handles these
- Long explanations or justifications - keep it terse and directive
- Output examples - they over-constrain the rewriter

### Length

Aim for 2-10 sentences. Usually 3-6 is the sweet spot. The example below is 6 sentences.

### Writing Style

Write it conversationally. The rewriter picks up the tone of the instruction itself, so if you want punchy output, write the instruction in punchy sentences. Avoid bullet lists or numbered rules - prose works better here.

### Example (Ken AI)

```
Make it sound as cool as possible. Show you've researched them. Use shorter sentences like I'm doing now. Better to have 2 shorter punchy sentences instead of 1 long boring one. Use power words. Sound human.
```

Notice: 6 sentences, conversational, hits voice + sentence structure + research signaling, no rules duplicated from defaults.

### Source Material

Pull voice/tone cues from:
- `{workspace}/research.md` - Tone preferences, voice guidelines, how the client talks about themselves
- `{segment_folder}/emails_v2.md` (or `emails.md`) - Style modeled in the sequence itself
- `{workspace}/notes.md` - Explicit style preferences from the client

### When to Omit

If the campaign has no distinctive voice/flow requirements, omit the `## Rewriting Instructions` H2 section entirely. The backend defaults are strong. But in practice, nearly every campaign benefits from 2-3 sentences of steering - default to including it.

### How It Gets Used

The `## Rewriting Instructions` H2 in `prompts.md` is parsed by the `campaign-configuration` skill's plan parser and pushed as a `Flow.Rewriting` system prompt via `PUT /v1/prompt-library`. At runtime the backend composes:
1. Default Rewriting Prompt (core principles, email flow, style, uniqueness)
2. Spam & Deliverability Rules
3. Company Context + Audience Description
4. Formatting Instructions
5. Email Sequence Context
6. **Your steering paragraph** (marked as higher-priority than the default)

## Review Phase - Live Test Details

This section documents the API-call half of the unified Review Phase (Task 3(b)). The text-review half lives in `review-workflow.md`.

### When the Review Phase Runs

The Review Phase (both halves) runs when **all** of these hold:
- `{plan_folder}/configuration.json` exists
- `configuration.json.campaign_id` is set (non-null) - the campaign has been pushed to Ken AI by `campaign-configuration`
- User has not passed `--skip-review` (alias: `--skip-testing`)

The Review Phase is **SKIPPED** (both halves) when:

- `--test-only` mode is specified (tests existing prompts, still gated on campaign_id)

### Step 1: Build Combined Prompt

Combine all prompts into a single `prompt_text` for the API:

```markdown
[System prompts from prompts.md]
- Instructions
- Company Context
- Rules
- Email Sequence

---

[User prompts with section headers]

### Subject Line
[Full prompt text for subject line]

### First Line
[Full prompt text for first line]

### PS Line
[Full prompt text for PS line]
```

The API parses `### Section Name` headers to map which variable generated which output.

**File Reference Resolution**: If the Email Sequence section contains a file reference like `[path/to/emails_v2.md]` (or `[path/to/emails.md]` as fallback), read the referenced file and replace the placeholder with the actual email content before building the combined prompt. The AI needs the full email sequence to generate contextually appropriate personalization.

### Step 2: Resolve segment_id

```
key = basename({segment_folder})                 # e.g. "1 - security-it"
seg = configuration.json.segmentation.segments[key]
segment_id = seg.segment_id
             if seg and seg.segment_id is not None
```

### Step 3: Run Personalization Test

Call `api_campaign_prompt_test`:

```python
api_campaign_prompt_test(
    campaign_id=<from configuration.json>,
    flow="Personalization",
    sample_size=5,
    segment_id=<resolved above>,
)
```

The backend auto-samples 5 personalization-ready contacts from the segment (falls back to other active contacts in that segment if the ready pool is empty). **Rewriting runs automatically in the same call when the campaign has it enabled** - no separate rewriting call is needed.

**Response structure:**
```json
{
  "usageSummary": {
    "averageInputTokens": ...,
    "averageOutputTokens": ...,
    "estimatedCost": ...
  },
  "samples": [
    {
      "contactId": ...,
      "fullName": "...",
      "linkedInUrl": "...",
      "status": "success" | "failure",
      "provider": "...",
      "model": "...",
      "rewritingProvider": "...",
      "rewritingModel": "..."
    }
  ],
  "personalizationOutputs": [
    {"contactId": ..., "name": "Subject Line", "response": "..."}
  ],
  "rewritingOutputs": [ ... ],
  "finalOutputs":    [ ... ]
}
```

- `finalOutputs` is what reaches the contact - use it for evaluation. When rewriting didn't run, `finalOutputs == personalizationOutputs`.
- Map outputs back to variable names by their `name` field (matches `### Section Name` headers in the combined prompt).

### Step 4: Evaluate Results

For each contact's `finalOutputs`, evaluate against these criteria:

| Criterion | Question | Check |
|-----------|----------|-------|
| Natural | Does it sound like a human wrote it? | No robotic patterns |
| Contextual | Does it fit the email sequence flow? | Matches tone/style |
| Relevant | Does it connect to the contact's actual data? | References their LinkedIn/company |
| Constrained | Does it follow length/format rules? | Word counts, no emojis |
| Fallback | If data was missing, did fallback work? | No blank outputs |

**Scoring:**
- **Pass**: 5/5 samples produce acceptable output
- **Minor Issues**: 3-4 samples pass, 1-2 have issues
- **Major Issues**: <3 samples pass or systemic problems

### Step 5: Auto-Revision Loop

If issues found:

1. **Identify patterns** across failures
2. **Apply targeted fixes** (see Auto-Revision Logic)
3. **Re-call `api_campaign_prompt_test`** - backend samples fresh contacts automatically
4. **Max 3 rounds total**

If still failing after 3 rounds: output prompts with warning, recommend manual review.

## Flow-Specific Testing

All flows use the same endpoint: `api_campaign_prompt_test(campaign_id, flow=..., sample_size=..., segment_id=...)`. Same gate applies - all flows require `configuration.json.campaign_id` to be set.

### Personalization (`flow="Personalization"`)

Standard flow for most campaigns. Tests:
- Subject lines
- First lines
- PS lines
- Body variables

Rewriting is invoked automatically in the same call when the campaign has rewriting enabled. `rewritingOutputs` and `finalOutputs` are populated alongside `personalizationOutputs`.

**Best practices for what goes through rewriting** (controlled by `to_rewrite` on each user prompt):
- Subject lines: `to_rewrite=0` (too risky)
- First lines, PS lines, body content: `to_rewrite=1`

### Qualification (`flow="Qualification"`)

Tests whether AI correctly qualifies/disqualifies contacts.

**Validation approach:**
1. `api_campaign_prompt_test(campaign_id, flow="Qualification", sample_size=5)` - omit `segment_id` (qualification is campaign-wide)
2. Read full contact data from database
3. Cross-reference: Does the qualification decision make sense?
4. Allow some oversight - qualification is assistive, not definitive
5. If qualification accuracy is weak on complex criteria, bump to a stronger model from `api_ai_supported_models(flow=1)` (e.g. `gpt-5.4-nano` or `gpt-oss-120b`) rather than the cheap default - don't hardcode an id, read the allowed list

### Segmentation (`flow="Segmentation"`)

Tests whether AI correctly categorizes contacts into segments.

**Validation approach:**
1. `api_campaign_prompt_test(campaign_id, flow="Segmentation", sample_size=5)` - omit `segment_id` (segmentation classifies *into* segments)
2. Read contact/company data from database
3. Verify segment assignment matches contact attributes
4. Check for edge cases between segments

## Evaluation Criteria

### Issue Detection Patterns

| Issue | Detection | Severity | Auto-Fix |
|-------|-----------|----------|----------|
| Generic output | "I noticed you work at a company" pattern | Major | Add explicit data source references |
| Wrong data source | References news, funding, events | Major | Remove invalid references |
| Too long | Subject > 6 words, first line > 20 words | Minor | Add stricter length constraints |
| Forced connection | "We went to same school" pattern | Major | Add "Never assume connections" rule |
| Missing fallback | Blank or error response | Major | Add explicit fallback text |
| Repetitive | Same phrase for 3+ contacts | Minor | Add "vary your approach" instruction |
| Off-tone | Mismatch with email sequence style | Minor | Reference email tone more explicitly |
| "I noticed/saw" | Starts with "I noticed you" or "I saw that" | Minor | Add "Never start with 'I noticed'" rule |

### Quality Thresholds

For each variable type:

| Variable | Max Length | Required Elements |
|----------|------------|-------------------|
| Subject Line | 6 words (ideally 3-4) | Specific detail, no emojis |
| First Line | 20 words | Hook, personalization, no greeting |
| PS Line | 2 sentences | Personal detail, starts with "PS" |
| Body Variable | Varies | Relevant to context |

## Auto-Revision Logic

When testing reveals issues, automatically revise prompts.

### Revision Decision Tree

```
Test Results
    |
    +-- All Pass (5/5) --> Output final prompts
    |
    +-- Minor Issues (3-4/5 pass)
    |       |
    |       +-- Identify specific prompt causing issue
    |       +-- Apply targeted fix
    |       +-- Re-test with 5 NEW contacts
    |
    +-- Major Issues (<3/5 pass or systemic)
            |
            +-- Analyze pattern across failures
            +-- Rewrite affected prompts
            +-- Re-test with 5 NEW contacts
```

### Common Fixes

| Issue Pattern | Fix to Apply |
|---------------|--------------|
| Generic output | Add: "Reference a specific detail from their LinkedIn profile, such as their current role, past experience, or skills listed" |
| Too long | Add: "MUST be under X words. Count your words before outputting." |
| Forced connections | Add to Rules: "Never start with 'I noticed' or 'I saw' - instead state observations directly" |
| Missing fallback | Add: "If no specific detail found, use this fallback: [generic but relevant text]" |
| Wrong data source | Remove invalid reference, substitute with valid source |
| Repetitive | Add: "Vary your approach for each contact. Don't use the same structure or phrases repeatedly." |

### Revision Round Tracking

Track testing rounds to prevent infinite loops:

```json
{
  "testing_rounds": [
    {
      "round": 1,

      "pass_rate": "3/5 (60%)",

        {"prompt": "Subject Line", "issue": "too_long", "count": 2}
      ],
      "revisions_made": [
        {"prompt": "Subject Line", "change": "Added word count constraint"}
      ]
    }
  ],
  "final_status": "pass"
}
```

## User Control Options

### Review Phase Flags

| Flag | Effect |
|------|--------|
| `--skip-review` (alias `--skip-testing`) | Skip the Review Phase entirely (no text review, no live test) |
| `--test-only` | Run the Review Phase on existing prompts without regenerating (still gated on `campaign_id`) |
| `--verbose` | Show full output for each sampled contact |
| `--compact` | Minimal output, just pass/fail summary |
| `--rounds N` | Set max review rounds (default: 3) |
| `--sample-size N` (alias `--contacts N`) | Set `sample_size` for `api_campaign_prompt_test` (default: 5, allowed 5-100) |

### User Approval Points

**Auto-proceed (no approval needed):**
- Minor fixes (length constraints, small wording changes)
- Pass on first round

**Pause for approval:**
- Major rewrites (changing data sources, restructuring prompts)
- 3rd round reached with persistent issues
- Campaign not configured on Ken AI (skip path confirmed to user)

## Review Phase Error Handling

| Error | Action |
|-------|--------|
| Campaign not configured on Ken AI (missing `configuration.json` or `campaign_id` is null) | Skip the Review Phase entirely (both text review and live test), output `prompts.md` with Graceful Degradation notice |
| All `samples[].status == "failure"` in the response | Do NOT output prompts - surface the failures and require user review |
| API error (rate limit) | Wait 30 seconds, retry once, then skip with warning |
| API error (auth / MCP disconnected) | Check MCP connection, provide troubleshooting steps |
| Timeout | Retry once with smaller `sample_size` (e.g. 5 → 3), then skip with warning |
| Endpoint returns 4xx for `campaign_id` not found | Same as "campaign not configured" - skip with notice |

### Graceful Degradation

If the Review Phase cannot run:

```markdown
## Review Skipped

Reason: [campaign not yet configured on Ken AI | explicit --skip-review | API error]

Prompts have been generated but NOT reviewed or tested against real contact data.

To enable review and testing:
1. Run `campaign-configuration` first to push the campaign to Ken AI
2. Re-run prompt-writer (or just the review portion) - the Review Phase will run automatically once `configuration.json.campaign_id` is set

Before deploying without review:
1. Manually check prompts for data source validity
2. Test with a few contacts in the Ken AI dashboard
3. Monitor early campaign performance closely

Prompts saved to: {segment_folder}/prompts.md
```

## Edit Mode (Improvement Context)

When invoked with improvement context (`mode: "edit"`), this skill edits existing prompts.md rather than generating new prompts.

### Detecting Edit Mode

Check for improvement context in the conversation:
- `mode: "edit"` indicates edit mode
- `trigger: "analytics"` or `trigger: "feedback"` indicates source
- `improvement_instruction` contains specific changes to make

### Edit Mode Workflow

1. **Read improvement context**: Extract `improvement_instruction` object
2. **Load existing prompts.md**: Read current prompts from campaign folder
3. **Identify scope**: Which prompts to modify (see scope table below)
4. **Apply targeted changes**: Only modify specified prompts
5. **Generate diff**: Show before/after comparison
6. **Get approval**: Wait for user to confirm changes
7. **Save**: Update prompts.md with approved changes
8. **Log**: Update improvement-log.md in plan folder

### Handling Edit Scopes

| Scope | Action | What Changes |
|-------|--------|--------------|
| "subject_line prompt" | Modify subject line user prompt only | Single prompt |
| "first_line prompt" | Modify first line user prompt only | Single prompt |
| "ps_line prompt" | Modify PS line user prompt only | Single prompt |
| "system prompts" | N/A - system prompts are auto-generated by the backend | No action needed |
| "all user prompts" | Rewrite all output-generating prompts | All user prompts |
| "add variable" | Add new AI variable prompt | New prompt added |
| "remove variable" | Remove existing prompt | Prompt removed |

### Edit Mode Output Format

```markdown
## Proposed Changes to prompts.md

### Subject Line Prompt

**Before:**
Write a very short subject line that grabs attention in fewer than 6 words...

**After:**
Write a curiosity-driven subject line under 5 words. Reference their company name or industry. Do not use AI personalization placeholders. Example: "Quick q about [company_name]"...

**Rationale:** Current AI-generated subject lines may be causing fatigue based on Email 3 performance

---

Approve these changes? [Yes/No/Modify]
```

### Preserving Context During Edits

When editing:
- **Keep unaffected prompts intact**: Only modify prompts in scope
- **Maintain prompt structure**: Keep Location, Email(s), format
- **Preserve variable names**: Names must match strategy.md
- **Keep data source constraints**: Never add invalid sources

### Analytics-Informed Editing

When `analytics_context` is provided:
- Check which emails use the underperforming prompts
- Reference `sequence_stats` to identify problem areas
- Consider if personalization is helping or hurting

### Feedback-Informed Editing

When `feedback_context` is provided:
- "AI feels off" = Adjust prompt constraints and fallbacks
- "Too generic" = Add more specific data source references
- "Forced connections" = Strengthen "don't assume" rules

### Edit Mode Checklist

Before saving edits, verify:
- [ ] Only specified prompts were modified
- [ ] Variable names match strategy.md exactly
- [ ] Data source constraints respected
- [ ] Prompt structure preserved
- [ ] Diff shown and approved by user
